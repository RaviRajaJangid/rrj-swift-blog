Swift Concurrency: async/await deep dive

📅 April 3, 2026 ⏱ 8 min read 🏷 Swift Concurrency: async/await deep dive

Swift Concurrency landed in Swift 5.5 and fundamentally changed how we write asynchronous code on Apple platforms. After years of callback hell, delegate soup, and the occasional Combine pipeline that reads like an abstract art piece, async/await gave us something we didn't know we desperately needed: asynchronous code that reads like synchronous code. But using it well in production is a different story from understanding the syntax. This post is the first in a series aimed at senior developers who want to go beyond the WWDC highlights and understand the machinery underneath.

The Suspension Model: What async/await Actually Does

The most important mental model shift is understanding that await is not a blocking call. When a function hits an await expression, it suspends — it gives up its thread so the runtime can schedule other work. This is radically different from locking a thread with a semaphore or a completion-handler chain sitting on a dispatch queue.

Consider what happens under the hood with a simple network call:

func fetchUser(id: String) async throws -> User {
    let url = URL(string: "https://api.example.com/users/\(id)")!
    let (data, response) = try await URLSession.shared.data(from: url)

    guard let httpResponse = response as? HTTPURLResponse,
          httpResponse.statusCode == 200 else {
        throw APIError.invalidResponse
    }

    return try JSONDecoder().decode(User.self, from: data)
}

When execution reaches try await URLSession.shared.data(from: url), the function suspends. The thread is freed. No thread is being held hostage waiting for network I/O. When the response arrives, the Swift concurrency runtime resumes the function — potentially on a different thread, though the actor model and main-actor annotations give us guarantees about which executor resumes us. This cooperative threading model is why Swift concurrency can scale to massive numbers of concurrent tasks without the thread-explosion problem that plagued GCD.

A function marked async can only be called from another async context or from a Task. This is intentional. It enforces a clear boundary between your synchronous and asynchronous worlds, and it prevents you from accidentally blocking a thread from a sync context.

Structured Concurrency: Task Groups and Child Tasks

One of the most underappreciated parts of Swift Concurrency is the structure it imposes on concurrent work. With GCD, you could fire off a dozen DispatchQueue.async calls and the compiler had no idea what you were doing. Cancellation was manual. Lifetimes were implicit. Errors disappeared into the ether.

Structured concurrency changes this. Every Task has a parent, and the parent doesn't complete until all of its children complete. This gives you automatic cancellation propagation and sane error semantics.

func loadDashboard(for userID: String) async throws -> Dashboard {
    async let profile = fetchProfile(userID: userID)
    async let feed = fetchFeed(userID: userID)
    async let notifications = fetchNotifications(userID: userID)

    // All three requests are in flight concurrently.
    // If any one throws, the others are automatically cancelled.
    return try await Dashboard(
        profile: profile,
        feed: feed,
        notifications: notifications
    )
}

The async let pattern is the lightweight version of a task group. All three network calls start immediately and run concurrently. The try await at the bottom waits for all three. If fetchProfile throws, Swift automatically cancels the other two in-flight tasks. You don't write a single line of cancellation logic.

For dynamic concurrency — where you don't know at compile time how many tasks you need — reach for withThrowingTaskGroup:

func fetchAllUsers(ids: [String]) async throws -> [User] {
    try await withThrowingTaskGroup(of: User.self) { group in
        for id in ids {
            group.addTask { try await fetchUser(id: id) }
        }

        var users: [User] = []
        for try await user in group {
            users.append(user)
        }
        return users
    }
}

The group automatically propagates cancellation inward and waits for all children before returning. Notice that iterating the group with for try await gives you results as they complete — not in insertion order. If order matters, return a tuple (index, User) and sort afterward.

One subtlety that bites teams in production: adding a task to a group after the group's body returns is a compile-time error. The structured lifetime is enforced by the type system, not convention. This is Swift Concurrency doing its job.

Actors: Isolation Without Locks

Before actors, protecting shared mutable state meant manual synchronization — DispatchQueue barriers, NSLock, os_unfair_lock. All of them correct when used carefully, all of them silent time bombs when used carelessly.

Actors give you data isolation as a language-level guarantee. The compiler won't let you read or mutate actor-isolated state from outside the actor without crossing an await boundary.

actor UserCache {
    private var store: [String: User] = [:]

    func user(for id: String) -> User? {
        store[id]
    }

    func insert(_ user: User, for id: String) {
        store[id] = user
    }
}

Accessing cache.user(for:) from outside the actor requires await. No await, no access — the compiler enforces it. There are no locks in your code because the actor's executor serialises access for you.

Where senior devs run into trouble: actor reentrancy. An actor suspends at every await, and while it's suspended another caller can enter. This means state you read before an await may be stale by the time the function resumes.

actor ImageLoader {
    private var cache: [URL: UIImage] = [:]

    func image(for url: URL) async throws -> UIImage {
        if let cached = cache[url] { return cached }

        // ⚠️ Actor is suspended here. Another caller can enter.
        let image = try await downloadImage(from: url)

        // cache[url] might already be set by the time we resume.
        // This is safe (just redundant work), but in other contexts
        // it could be a logical race.
        cache[url] = image
        return image
    }
}

Reentrancy isn't a bug — it's a deliberate design choice that prevents deadlocks. The fix is to check your invariants again after every await, or restructure to avoid the interleaving entirely.

@MainActor: The UI Thread Contract

@MainActor is just a global actor that runs on the main thread. Marking a type or function with it gives you a compiler-enforced guarantee that code runs on the main queue — no more DispatchQueue.main.async sprinkled through your view models.

@MainActor
final class FeedViewModel: ObservableObject {
    @Published var posts: [Post] = []
    @Published var isLoading = false

    func loadPosts() async {
        isLoading = true
        defer { isLoading = false }

        do {
            // Suspends here — jumps off main thread for the network call
            let fetched = try await postService.fetchLatest()
            // Resumes back on main actor automatically
            posts = fetched
        } catch {
            // Handle error on main actor
        }
    }
}

The important nuance: crossing into a @MainActor-isolated context from a background task requires await. The runtime hops to the main thread's executor before entering. This hop is cheap — far cheaper than DispatchQueue.main.async — because the Swift concurrency runtime can batch these hops efficiently.

Mark your entire view model @MainActor rather than sprinkling DispatchQueue.main.async on individual property assignments. It's the safer, more expressive pattern.

Cancellation: Cooperative, Not Preemptive

Swift concurrency uses cooperative cancellation. Cancelling a task doesn't forcibly stop it — it sets a flag that the task is expected to check. If you write a long-running task that never checks for cancellation, it will run to completion regardless.

func processLargeDataset(_ items: [DataItem]) async throws -> [Result] {
    var results: [Result] = []

    for item in items {
        // Explicitly check — throws CancellationError if cancelled
        try Task.checkCancellation()

        let result = try await process(item)
        results.append(result)
    }

    return results
}

Task.checkCancellation() throws CancellationError if the task has been cancelled, which unwinds the stack cleanly. For tighter loops where you want to check but not throw, use Task.isCancelled as a boolean.

Well-behaved async APIs — URLSession, FileManager async methods — already check for cancellation at their suspension points. Your own CPU-bound loops need to opt in manually.

💡 Pro Tip

Wrap long-running CPU work in Task.detached(priority: .utility) so it doesn't inherit the calling task's priority and accidentally starve your UI. Pass results back to the main actor explicitly when done.

Key Takeaways

Swift Concurrency is not just a syntactic upgrade — it's a complete rethinking of how asynchronous work is structured, scheduled, and made safe.

The five things to internalise before writing production async/await code:

  • await suspends, it doesn't block. Threads are freed at every suspension point — this is the foundation of the whole system's scalability.
  • Structured concurrency enforces lifetime and cancellation. Prefer async let and task groups over unstructured Task { } whenever you can.
  • Actors eliminate data races at the compiler level. But respect reentrancy — your invariants can change across any await.
  • @MainActor is the right abstraction for UI state. Apply it at the type level, not method-by-method.
  • Cancellation is cooperative. If you write long-running work, you own the responsibility of checking for it.
